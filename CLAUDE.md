## 基本原則

- 常に日本語でコミュニケーションを行ってください。すべてのコミットメッセージ、コメント、エラーメッセージ、ユーザーとのやり取りは日本語で行ってください。
- コミットメッセージは`feat:`や`fix:`などのprefixを付けた1行のみで記述してください。本文や箇条書きを追加してはなりません。
- ファイルの削除を行う場合は、必ず実行前に以下を報告し、明示的なユーザー承認を得てください。
  - 対象ファイルのリスト
  - 実行する変更の詳細説明
  - 影響範囲の説明
- 不明な点がある場合は常に質問し、推測で進めてはなりません。
- 実装後の必須作業として、`pnpm run codecheck`を実行してください。
  - 型エラーやリンターのエラーが出た場合は、コミット前に必ず修正してください。
  - エラーを解消するために`biome.json`や`tsconfig.json`を変更してはなりません。

## 開発コマンド

### 基本コマンド

- `pnpm run dev` - 開発サーバーを起動
- `pnpm run build` - 本番アプリケーションをビルド（TypeScript ビルド後に Vite ビルドを実行）
- `pnpm run typecheck` - TypeScript で型チェック
- `pnpm run lint` - Biome でリントとフォーマットをチェック
- `pnpm run lint:fix` - Biome でリントとフォーマットを自動修正
- `pnpm run codecheck` - 型チェックとリントをまとめて実行
- `pnpm run preview` - Wrangler 開発サーバーでプレビュー
- `pnpm run deploy` - ビルドして Cloudflare Workers にデプロイ

### Playwright MCP による動作確認

- Playwright MCP で撮影したスクリーンショットは、必ず `.playwright-mcp/` ディレクトリに保存してください(`filename` に `.playwright-mcp/xxx.png` のようにディレクトリ付きで指定します)。
- `.playwright-mcp/` は gitignore 済みです。プロジェクトルート直下にスクリーンショット等の検証用ファイルを作成しないでください。

## アーキテクチャ

### 技術スタック

- **言語**: TypeScript
- **フレームワーク**: React 19 + Vite
- **ルーティング**: React Router
- **エディタ**: Tiptap
- **データ保存**: Dexie (IndexedDB)
- **スタイリング**: CSS Modules
- **コード品質**: Biome
- **デプロイ**: Cloudflare Workers (Wrangler)

### プロジェクト構造

```bash
src/
├── App.tsx             # ルートコンポーネント・ルーティング定義
├── App.module.css      # アプリ全体のスタイル
├── Sidebar.tsx         # メモ一覧サイドバー
├── MemoEditor.tsx      # Tiptap エディタと文字数カウント
├── MemoPage.tsx        # メモ表示・編集ページ
├── NewMemoPage.tsx     # メモ新規作成ページ
├── db.ts               # Dexie (IndexedDB) の設定
├── index.css           # グローバルスタイル
├── main.tsx            # エントリーポイント
└── worker.ts           # Cloudflare Workers エントリーポイント
```

### インポートとパスエイリアス

- パスエイリアス `@/` が `src/` にマップされています（`tsconfig.json` の `paths` で定義）。
- 同一ディレクトリ内のインポートでは相対パスを使用して構いません。それ以外は `@/` から始まるパスエイリアスを使用してください。

## コーディングガイドライン

### `any`の禁止

- いかなる理由があっても`any`を使用してはなりません。
- `unknown`や`never`の使用も避けてください。
- 実データと一致する型を定義してください。

### 型アサーションの禁止

- 型アサーションは禁止です。
- 型アサーションを使用する場合は、明確な理由をコメントアウトとして記述してください。

### `interface`の禁止

- 型定義に`interface`を使用してはなりません。`type`を使用してください。

### コメントの禁止

- 原則としてコメントは記述してはなりません。
- 型アサーションやuseEffectの使用理由など、他のガイドラインが記述を求める場合のみ例外とします。
- コメントを書く場合は括弧を使用しないでください。

### 過度な抽象化の禁止

- 無駄に関数化・定数化しすぎてはなりません。
- 再利用される明確な根拠がない限り、処理の切り出しや定数への抽出を行わないでください。
